# 开发工作流方法论

与 Claude Code 协作开发的完整工作流。

---

## 一、工作流总览

```
进入项目 → 了解现状 → 规划 → 实现 → 测试 → (审计) → 提交 → 迭代
              ↑                                         |
              └── 文档落后? → /catch-up ────────────────┘
```

**核心循环**：每次对话（无论规划、实现还是测试）结束时，如有文件变更 → /update-docs → commit。

---

## 二、阶段详解

### 1. 进入项目（Context Loading）

| 步骤 | 动作 | 产出 |
|------|------|------|
| 自动 | 加载 `CLAUDE.md` | 项目概要、命令、规则 |
| 按需 | 读 `record/project/index.md` | 功能全景、状态总览 |
| 按需 | 读相关 `features/fXX-*.md` | 功能详情（状态 + 变化历史） |
| 如需 | `/catch-up` | 文档与代码同步 |

**何时 /catch-up**：首次建立文档、其他人改了代码没更文档、切换到新分支后文档缺失。

### 2. 规划（Planning）

1. 进入 Plan Mode，设计实现方案
2. 对话结束时 → /update-docs → commit
   - 新功能：新建 feature 文件，「状态」节写计划，「变化」节添加 `[计划]` 条目
   - 已有功能：「变化」节添加 `[计划]` 条目
   - index.md 增量更新对应行

### 3. 实现（Implementation）

1. 按 plan 写代码（PostToolUse hook 自动 git add）
2. 对话结束时 → /update-docs → commit
   - 「状态」节刷新为最新实现状态
   - 「变化」节添加 `[实现]`/`[修改]`/`[修复]` 等条目
   - 包含代码修改描述 + 关联测试结果

### 4. 测试（Testing）

1. 运行测试（Claude 执行 或 用户粘贴结果）
2. 对话结束时 → /update-docs → commit
   - 测试结果记入当前变化条目（或新建条目）
   - 跨功能测试：在每个涉及的功能文件中都记录

### 5. 审计（Auditing，可选）

- `/audit` — 全局审计
- `/audit F01` — 审计特定功能
- `/audit F01 F03` — 审计多个功能

输出优先级排序的待办清单，更新对应文档。

### 6. 提交（Committing）

- 每次对话结束，如有文件变更 → /update-docs → commit
- 一条变化记录 ≈ 一次 commit
- commit message 根据文档变化生成，准确描述更改
- 多个小 commit 可用 `git rebase -i` 合并为一次大更新

## 三、场景速查表

| 我想做什么 | 用什么 |
|-----------|--------|
| 首次进入项目 | CLAUDE.md → `/catch-up` |
| 开始新功能 | Plan Mode → `/update-docs` → commit |
| 写完代码 | `/update-docs` → commit |
| 运行了测试 | `/update-docs` → commit |
| 用户粘贴测试结果 | `/update-docs` → commit |
| 文档落后了 | `/catch-up` |
| 检查代码质量 | `/audit` 或 `/audit FXX` |
| 寻找改进机会 | `/suggest` 或 `/suggest FXX` |
| 发现/修复 Bug | 写在变化条目中 → `/update-docs` → commit |

## 四、文档体系

```
项目根/
├── CLAUDE.md                          ← 项目入口
└── record/project/
    ├── index.md                       ← 功能索引 (hub)
    ├── workflow.md                    ← 本文件（方法论）
    └── features/                      ← 功能详情 (spokes)
        └── fXX-name.md
```

### 分支与文档

`record/project/` 跟随 git 分支，每个分支有独立的文档状态。切换分支后文档自然切换，如果新分支缺少文档 → `/catch-up`。

### index.md

功能索引，增量更新（只改变化的行，不重新生成）。

包含：架构概览 + 功能清单表 + 全局问题汇总。

## 五、功能划分原则

一个功能 = 一组协同完成同一职责的代码。

| 规则 | 说明 |
|------|------|
| 按逻辑功能，不按文件 | 一个功能可跨多个文件 |
| 一个文件可属于多个功能 | 如含多个独立职责 |
| 避免过细 | 每个文件一个功能 |
| 避免过粗 | 整个模块一个功能 |
| 独立生命周期 | 总一起改的代码 → 合为一个功能 |

## 六、Skill 体系

| Skill | 用途 | 触发 |
|-------|------|------|
| `/catch-up` | 初始化/追赶文档 | 首次、文档落后、切换分支 |
| `/update-docs` | 日常更新 | 每次对话结束有变更时 |
| `/audit` | 审计质量问题 | 按需，支持全局或指定功能 |
| `/suggest` | 增量改进建议 | 按需，支持全局或指定功能 |

---

## 七、约定

| 项目 | 规范 |
|------|------|
| 时间戳 | UTC+8 北京时间，`YYYY-MM-DD HH:MM` |
| 功能 ID | FXX 递增，文件名 `fXX-kebab-case.md` |
| 实现状态 | ✅已完成 / 🔧进行中 / 📋计划中 / ❌已废弃 |
| 功能状态 | ✅无误 / ⚠️有注意事项 / 🔴有Bug |
| 维护状态 | 🟢在用 / 🟡冻结 / 🔴弃用 |
| 优先级 | P0 致命 / P1 高 / P2 中 / P3 低 |
| 变化 tag | `[计划]` `[实现]` `[修改]` `[修复]` `[重构]` `[弃用]` `[启用]` |
| 折叠 | 用 `<details><summary>` 保持文件可扫描 |
