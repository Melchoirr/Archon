# suggest

## 流程

1. **确定范围** — 无参数→全部功能；有参数→指定 FXX。从 index.md 获取功能列表。

2. **分析每个功能** — 读 features/fXX-*.md + 对应源文件，寻找增量改进机会：
   - 代码简化（可读性、冗余逻辑）
   - 性能优化（不必要的重复计算、可缓存的操作）
   - 可维护性（魔法值提取、接口统一、文档补充）
   - 功能增强（缺失但有价值的能力扩展）
   - 运行流程优化（冗余步骤、可并行的步骤）
   - 依赖解耦（循环依赖、过度耦合）
   - 数据流简化（不必要的数据变换、可合并的处理步骤）
   - 已有建议的状态检查（💡新提出 的是否仍然有效）

3. **输出建议清单**（直接展示给用户，由用户决定处理）：
   ```
   ## 增量建议 — YYYY-MM-DD (UTC+8)
   范围：全局 / F01, F03

   ### [FXX] 功能名称
   - 💡 建议标题 — 具体描述
   - 💡 建议标题 — 具体描述

   ### [FYY] 功能名称
   - 💡 建议标题 — 具体描述
   ```

4. **更新文档** — 将新建议写入对应 features/ 的「建议」节：
   ```markdown
   ## 建议
   - 💡新提出 — 建议内容 (YYYY-MM-DD)
   ```
   保留已有条目，不覆盖。已存在的相同建议不重复添加。

5. **Commit** — `git commit -m "docs(suggest): [范围] 新增 N 条增量建议"`

## 建议 vs 审计

| | `/suggest` | `/audit` |
|---|---|---|
| 性质 | 增量改进，锦上添花 | 问题检测，风险排查 |
| 紧迫度 | 无优先级，用户自行决定 | P0-P3 优先级体系 |
| 用户动作 | 采纳(→已计划) 或 否决 | 必须处理（至少评估） |

## 建议状态
- 💡新提出 — 刚生成，等待用户决定
- 📋已计划 — 用户采纳，已纳入实现计划
- 🚫否决 — 用户明确不采纳
