# audit

## 流程

1. **确定范围** — 无参数→全部功能；有参数→指定 FXX。从 index.md 获取功能列表。

2. **检查每个功能** — 读 features/fXX-*.md + 对应源文件，检查：
   - 测试覆盖（有无测试记录？最近结果？）
   - 未解决的问题
   - 代码风险（重复代码、缺少错误处理、硬编码值、边界条件）
   - 运行流程描述是否与代码一致（函数签名、调用链是否过时）
   - 依赖关系是否完整（上游/下游是否遗漏）
   - flow.md 与实际代码流程是否一致（涉及的阶段描述、mermaid 流程图）

3. **输出报告**：
   ```
   ## 审计报告 — YYYY-MM-DD (UTC+8)
   范围：全局 / F01, F03

   ### 🔴 P0-P1
   - [FXX] 问题 → 建议

   ### 🟡 P2
   - [FXX] 问题 → 建议

   ### ⚪ P3 待测试
   - [FXX] 功能 — 关键路径未验证
     建议：`测试命令`

   ### 📄 文档准确性
   - [FXX] 运行流程描述与代码不一致 → 需更新的具体内容
   - [FXX] 依赖关系遗漏 → 缺少的上游/下游
   - flow.md 阶段 N 与实际代码流程不一致 → 具体差异
   ```

4. **更新文档** — 问题写入对应 features/ 的状态节 + 变化节追加 `[审计]` 条目。增量更新 index.md 状态列。

5. **Commit** — `git commit -m "docs(audit): 审计 [范围] 发现 N 个问题"`

## 优先级
P0 数据丢失/不可用 · P1 核心功能受损 · P2 代码质量 · P3 改善建议
