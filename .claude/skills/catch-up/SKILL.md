# catch-up

## 流程

### 1. 探索项目结构

不预设任何目录名或框架，按以下步骤发现项目结构：

- `ls` 项目根目录，识别源码目录（排除 `.git`, `node_modules`, `__pycache__`, `.venv`, `dist`, `build` 等构建产物）
- 读项目元文件（README, pyproject.toml, package.json, Cargo.toml, go.mod 等），提取：项目名称、语言、依赖、入口命令
- `git log --format="%ai %H %s" -20` 获取近期提交历史
- 递归浏览源码目录（优先读入口文件、配置文件、模块 __init__.py），理解模块划分和职责
- 读现有 `record/project/` 文档（如有），对比代码判断哪些需要更新

### 2. 划分功能

按逻辑功能划分，非按文件。一个功能 = 协同完成同一职责的代码（可跨多文件）。

粒度判断：会被独立修改、独立测试的最小逻辑单元。避免过细（每文件一功能）或过粗（整模块一功能）。

### 3. 追踪运行流程

从入口文件开始，沿调用链追踪完整运行流程：

**全局流程**：
- 识别程序入口（main 函数、CLI 命令、HTTP server 启动等）
- 追踪主路径：入口 → 调度/路由 → 业务逻辑 → 输出
- 标注关键分支（条件跳转、状态机转换、循环/重试）
- 标注每一步涉及的功能 ID（FXX）

**局部流程**（每个功能）：
- 触发条件：谁调用、什么时候调用
- 内部处理步骤：核心算法/逻辑的简要描述
- 输出：返回什么、副作用（文件写入、API 调用等）
- 上下游：依赖哪些功能（上游）、被哪些功能依赖（下游）

### 4. 生成 flow.md

写入 `record/project/flow.md`，要求**详尽**——读者仅凭此文件就能理解系统如何运行：

```markdown
# 运行流程

## 全局流程图

（mermaid flowchart，节点标注 [FXX] + 关键文件名 + 简要职责）
（复杂分支用子图 subgraph 分组）

## 入口与初始化
- 程序如何启动（命令行参数、配置加载、环境校验）
- 初始化了哪些对象/资源
- 从源码提取具体参数和默认值

## 主流程阶段详解

### 阶段 1：阶段名称
- **涉及功能**：FXX, FYY
- **触发条件**：什么情况下进入此阶段
- **输入数据**：接收什么（数据结构、来源）
- **处理逻辑**：
  1. 第一步：具体做什么（调用哪个函数、关键参数）
  2. 第二步：...
  3. ...
- **输出数据**：产出什么（数据结构、写入哪里）
- **关键分支**：条件判断 → 不同走向（成功/失败/重试/跳过）
- **错误处理**：异常时怎么办

### 阶段 2：...
（同上格式）

## 状态管理与持久化
- 系统状态如何保存和恢复（文件、数据库、内存）
- 关键状态转换规则

## 数据流总览
- 从输入到输出，数据经过哪些变换
- 关键数据模型及其字段（从源码提取，标注定义位置）
```

### 5. 生成 features/fXX-*.md

每个功能一个文件，要求**详尽**——读者仅凭此文件就能理解该功能的全貌：

```markdown
# [FXX] 功能名称

## 状态
- **实现状态**：✅已完成 / 🔧进行中 / 📋计划中 / ❌已废弃

## 核心文件
- `path/file.py:行号` — `ClassName` / `function_name()`，职责描述
- `path/file2.py:行号` — ...
（列出所有关键文件，每个标注行号、类/函数名、职责）

## 功能描述
从源码提取的详细描述，不猜测。包含：
- 这个功能解决什么问题
- 核心概念和术语解释
- 关键数据结构（字段名、类型、用途）
- 配置项和默认值

## 运行流程

### 触发条件
- 谁调用（具体函数/模块）、什么时候调用
- 前置条件（需要什么数据/状态就绪）

### 处理步骤
1. **步骤名称** — 具体做什么
   - 调用 `function_name(params)` → 返回什么
   - 关键逻辑：条件判断、循环、算法要点
2. **步骤名称** — ...
3. ...

### 输出
- 返回值：数据结构和含义
- 副作用：文件写入、状态变更、API 调用

### 依赖关系
- **上游**：FXX（提供什么） / FYY（提供什么）
- **下游**：FXX（消费什么） / FYY（消费什么）

### 错误与边界情况
- 已知的异常处理逻辑
- 边界条件和限制

## 测试方法
可执行的命令或代码片段，附预期输出说明

## 建议
（暂无）

## 变化
### [实现] YYYY-MM-DD HH:MM — 初始实现 (`hash`)
- **目的**：功能目标
- **改动**：新增了什么（文件、函数、逻辑）
- **验证**：怎么验证正确性（命令 + 结果），未验证则写「未测试」
```

### 6. 生成/更新 index.md

- 架构概览：从代码探索结果生成（不硬编码），引用 `[运行流程](flow.md)`
- 功能清单：分点列表格式（非表格），每个功能：

```markdown
### FXX — 功能名称 · 状态 · 维护状态
- **核心文件**：`file1.py`, `file2.py`
- **上游**：FXX（提供什么） / **下游**：FYY（消费什么）
- **最后变更**：YYYY-MM-DD HH:MM

<details><summary>功能概要</summary>

**做什么**：一段话描述功能职责
**怎么做**：核心逻辑的简要说明（3-5 行）
**关键接口**：主要的类/函数及其签名
**数据流**：输入什么 → 经过什么处理 → 输出什么

</details>

→ [完整详情](features/fXX-name.md)
```

- 全局问题汇总：从 features/ 聚合

### 7. Commit

`git add record/ && git commit -m "docs(catch-up): 同步文档至当前代码状态"`

## 约定
- 时间：UTC+8，`YYYY-MM-DD HH:MM`。用 `git log --format="%ai"` 获取精确时间。
- ID：FXX 递增，读 index.md 最大 ID 后 +1。文件名 `fXX-kebab-case.md`。
- 不猜测实现细节，直接读源码。
- 状态：✅已完成 / 🔧进行中 / 📋计划中 / ❌已废弃
- 折叠：变化详情用 `<details><summary>` 包裹
